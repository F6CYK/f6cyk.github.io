# Objet

Le présent document dresse un état complet du développement du menu documentaire du site.

L'objectif n'est pas la réalisation d'un simple menu déroulant mais la mise en place d'un composant documentaire générique, destiné à devenir le mécanisme de navigation de l'ensemble du site.

La rubrique **Documentation** constitue le premier terrain d'expérimentation avant généralisation aux autres rubriques.

---

# Objectif recherché

Le menu documentaire doit être entièrement généré à partir d'une description de l'arborescence documentaire.

Le code Liquid ne doit contenir aucune connaissance de cette arborescence.

Toute évolution documentaire devra pouvoir être réalisée uniquement par modification des données.

---

# Principe retenu

L'architecture repose sur une séparation stricte des responsabilités.

- les données décrivent l'organisation documentaire ;
- Liquid construit récursivement le menu ;
- le CSS assure uniquement la présentation.

Cette séparation est considérée comme un choix d'architecture majeur.

---

# Organisation des fichiers

À ce stade du développement, les fichiers concernés sont :

```text
_data/
└── documentation.yml

_includes/
├── nav.html
└── recursive-menu.html

assets/
└── css/
    ├── layout.css
    └── components.css
```

---

# Organisation des données

L'arborescence documentaire est décrite exclusivement dans :

```text
_data/documentation.yml
```

Le modèle retenu est récursif.

```yaml
- titre:
  url:
  enfants:
```

Les propriétés `url` et `enfants` sont facultatives.

Ce modèle est validé.

---

# État d'avancement

Les éléments suivants sont validés.

- modèle de données récursif ;
- génération récursive du menu par Liquid ;
- intégration du menu dans la navigation principale ;
- ouverture récursive des différents niveaux.

Le fonctionnement général est désormais obtenu.

---

# Fonctionnement actuel

Le menu permet actuellement de parcourir correctement une hiérarchie documentaire.

L'exemple de référence est actuellement :

```text
Documentation
└── Composants
    └── Circuits intégrés
        └── XR2206
```

Chaque niveau est correctement généré à partir des données.

La récursivité du composant est donc considérée comme acquise.

---

# Difficultés rencontrées

Les difficultés ne concernent plus le fonctionnement du composant.

Elles concernent désormais exclusivement sa présentation.

Deux feuilles de style interviennent actuellement :

```text
assets/css/layout.css
assets/css/components.css
```

Le partage des responsabilités entre ces deux fichiers n'est pas encore suffisamment clarifié.

Plusieurs modifications successives ont conduit à un comportement parfois instable du menu.

Aucune évolution supplémentaire ne devra être entreprise avant une analyse complète de cette organisation.

---

# Choix d'architecture validés

Les décisions suivantes sont désormais considérées comme acquises.

- utilisation d'une structure documentaire récursive ;
- séparation stricte entre données, génération et présentation ;
- génération complète du menu par Liquid ;
- absence d'éléments codés en dur ;
- réutilisation future du composant pour les autres rubriques du site.

Ces choix ne sont plus remis en cause.

---

# Retour d'expérience

Le développement a montré qu'une approche consistant à modifier progressivement les feuilles de style conduit rapidement à des régressions difficiles à analyser.

Le temps de déploiement observé sous GitHub Pages est actuellement de l'ordre de dix à onze minutes.

Ce délai rend les expérimentations successives particulièrement coûteuses.

Il apparaît désormais préférable de préparer des évolutions complètes et cohérentes avant toute publication.

---

# Philosophie de conception

Le menu documentaire constitue un composant générique.

Son comportement doit dépendre exclusivement de l'arborescence documentaire.

L'ajout, la suppression ou la réorganisation d'une branche documentaire ne devront jamais nécessiter de modification du code Liquid.

À terme, ce composant devra pouvoir être réutilisé sans adaptation dans les différentes rubriques du site.

---

# Évolutions prévues

Les travaux restant à réaliser concernent exclusivement la présentation.

Les objectifs sont les suivants :

1. stabilisation complète du comportement CSS ;
2. amélioration de la lisibilité des différents niveaux ;
3. présentation des sous-menus sous forme d'arborescence visuelle ;
4. mise en évidence des branches possédant des descendants ;
5. gestion de l'élément actif dans l'arborescence.

Aucune évolution fonctionnelle supplémentaire n'est envisagée avant la stabilisation de la présentation.

---

# Méthode de développement

La méthode de développement retenue est désormais la suivante.

Chaque évolution devra :

1. faire l'objet d'une analyse complète préalable ;
2. être livrée sous forme d'une modification unique et cohérente ;
3. être validée après publication ;
4. être documentée.

Les modifications partielles ou expérimentales sont abandonnées.

Chaque proposition devra préciser :

- le fichier concerné ;
- son chemin complet ;
- l'action à réaliser ;
- le bloc complet à remplacer.

Cette méthode vise à limiter les régressions et à réduire le nombre de déploiements nécessaires.
